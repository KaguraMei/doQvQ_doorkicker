# 糯叽叽项目 - AI 提示词系统总结

## 概述
该项目是一个基于 AI 对话的社交应用，包含多个模块，每个模块都有特定的提示词注入系统。

--- 

## 1. 对话总结系统 (Summary System)

### 位置
- `aiSummaryService-Bx5aTYVV.js`

### 核心提示词功能

#### 1.1 对话总结提示词
**主要功能**: 将多条聊天消息总结为结构化摘要

**提示词结构**:
```
You are a dialogue analysis expert. Produce a concise, coherent summary...

**OUTPUT LANGUAGE: {语言}**

**IDENTITY MAP — CRITICAL:**
- 玩家「User」= 真实用户
- AI角色「Character」= AI扮演的角色
- 必须严格区分双方身份，不可混淆

**SUMMARY RULES:**
1. PERSPECTIVE: 第三人称视角
2. 压缩要点：核心信息、关键话题、情感转变
3. Voice-aware compression: 保留标志性用语、昵称、口头禅、语气词
4. 说话人归属：明确标注谁说的话
5. 时间保真：保留所有具体时间（周五下午3点、明天等）
6. 逻辑流畅：按时间或因果组织
7. 保留关键项：地点、承诺、计划、关系转折

**OUTPUT FORMAT:**
<summary body> (纯文本，无括号，~{字数}字)

[关键词: 词1, 词2, ...]
[属性: importance|emotion|topic1,topic2]
[悬念: type|content] OR [悬念: 无]
[立场: {角色}对{用户}的内心态度]
[成长-关系: ...] (可选)
[成长-自我: ...] (可选)
[事实: ...] OR [事实: 无]
[事实失效: ...] OR [事实失效: 无]
```

#### 1.2 情侣空间日记提示词
当启用情侣空间功能时，额外生成：

```json
[恋爱路程: [{
  "date": "ISO8601",
  "content": "角色第一人称日记 25-60字",
  "tag": "milestone|daily|dating...",
  "imageRef": "img_N或null",
  "reply": "引用原文",
  "replyFrom": "user|char|null"
}]]

[剧情成就: [{
  "title": "4-8字诗意标题",
  "desc": "简短描述",
  "color": "bronze|silver|gold|diamond",
  "icon": "图标名"
}]]
```

**关键规则**:
- 日记必须用「我」= 角色本人的视角
- 语气必须符合角色人设（霸总/温柔/毒舌/冰山等）
- 成就只在真正的里程碑事件时生成（~20%）

## 2. 快捷回复系统 (Quick Reply System)

### 位置
- `useQuickReply-CdHVvLEb.js`
- `userSuggestionPrompts-DFdKgzKs.js`

### 核心功能

#### 2.1 用户建议回复提示词
**预设方向标签**:
```javascript
{
  推劇情: "take initiative, make a key decision, or push the plot forward",
  談心: "open up emotionally, share inner feelings, memories, or vulnerability",
  轉移話題: "naturally steer toward a new topic, event, or shift the mood"
}
```

#### 2.2 单个回复生成提示词
```
[System instruction: This is an ONLINE INSTANT MESSAGING chat...

Direction: {方向} — {描述}

CRITICAL OUTPUT RULES:
- 这是即时通讯，不是小说
- 禁止动作描述、内心独白、旁白
- 不使用「」、*号、[]等小说符号
- 用自然的打字语气：简短、随意、碎片化
- 1-4个气泡，用换行分隔
- 每个气泡：~3-25个汉字
- 可使用emoji、重复字符（嗯嗯、哈哈哈）

Output "{用户名}"'s reply (multiple short lines):]
```

#### 2.3 多个回复生成提示词
```
Generate {数量} different short reply drafts...

Output format:
[1]
first bubble
second bubble
[2]
first bubble
second bubble
...
```

## 4. 心声生成系统 (Inner Voice / Xinsheng System)

### 位置
- `ChatDetailScreen-qTVdjKXv.js`

### 核心功能
AI 生成角色的"内心活动"JSON 卡片

**默认提示词结构**:
```
At the very END of every reply, append exactly ONE single-line JSON:
{"t":"xinsheng", ...}

Schema and fields follow the custom guidance...
```

**可自定义字段示例**:
- `innerVoice`: 内心独白
- `secret`: 秘密想法
- `mood`: 心情
- `energy`: 精力值

**特性**:
- 支持 CSS 自定义样式
- 支持完全自定义提示词
- 可设置延续字段（让 AI 看到上几次的心声值）
- 支持预设库随机套用

### 关系成长系统
当启用时注入：

```

依然{核心特质}，但对{用户}开始{微小行为变化}
```

## 10. 提示词优先级与覆盖规则

### 优先级顺序（从高到低）
1. **用户自定义覆盖** (customSummaryInstructions, xinshengCustomPrompt)
2. **身份映射与说话人归属** （结构性基础，不可覆盖）
3. **Voice-aware preservation** （保留双方语言特征）
4. **默认规则**

### 覆盖示例
```
⚠️ **USER REQUIREMENT — READ FIRST, HIGHEST PRIORITY:**
{用户自定义指令}

The above requirement takes precedence over all default rules below
(except identity/speaker attribution).
```

## 总结

该项目的提示词系统特点：

1. **模块化设计**: 不同功能有独立的提示词模板
2. **高度可定制**: 用户可以覆盖大部分默认提示词
3. **结构化输出**: 使用特定格式（JSON、标记语法）确保解析
4. **身份保护**: 严格区分玩家和AI角色，防止混淆
5. **语音保护**: 保留双方的语言特征和标志性用语
6. **时间保真**: 强调保留具体时间信息
7. **成长系统**: 支持关系和自我的微小变化追踪
8. **多语言支持**: 支持中文（简繁）、英语、日语

## 总结：完整的提示词流程

1. **收集上下文** → 构建 `_r` 参数对象（包含角色、用户、关系、记忆等所有信息）

2. **构建基础提示词** → `W(_r)` 函数生成基础系统提示词

3. **动态注入模块** → 按顺序追加：
   - 世界书内容
   - 记忆系统（心智模型、情节摘要、跨应用记忆）
   - MCP 上下文
   - HTML 模板
   - 头像系统
   - 音乐系统
   - 心声指令
   - Reroll 反馈
   - 手机活动检测

4. **构建消息数组** → 将系统提示词作为第一条消息，添加对话历史

5. **特殊消息处理** → 根据场景添加隐藏指令（表情包、拉黑状态、朋友圈等）

6. **发送 API 请求** → 完整的消息数组发送给 AI 模型

---

**关键特点**：

## 16. 应用反应系统 (App Reaction / Phone Activity)

### 位置
`appReactionGenerator-dTJW9WSt.js`

### 功能
生成角色对用户使用手机 APP 的反应台词（桌宠监控手机活动）

### 核心提示词

```
You write three sets of short reaction lines that ${角色名} would say 
when they catch ${用户名} using a phone app called "${应用名}".

CONTEXT
${角色名} is a desktop pet — a tiny floating character living on ${用户名}'s phone screen.
${角色名} can see what app ${用户名} is using right now.
You write THREE sets of lines for three different moments around "${应用名}":

[OPEN]      ${数量} lines — ${用户名} just opened "${应用名}". ${角色名} reacts in the moment.
[CLOSE]     ${数量} lines — ${用户名} just closed/switched away from "${应用名}". 
            ${角色名} comments on what they were doing.
[LONG_STAY] ${数量} lines — ${用户名} has been inside "${应用名}" for over an hour. 
            ${角色名} can't help saying something.

VOICE LOCK (most important)
After writing each line, silently ask: "If I deleted ${角色名}'s name, 
could ANY other character say this exact sentence?"
If yes → throw it out, rewrite with something only ${角色名} would say 
(their slang, their pet peeves, their callbacks to shared memory, 
their specific way of teasing/caring).
Every line must carry ${角色名}'s fingerprint.

CONTEXT YOU MUST USE
- ${角色名}'s persona below — quirks, hobbies, what they care about, how they talk
- ${用户名}'s persona — what THEY are into, why they would open "${应用名}"
- The relationship between them — intimacy level decides how possessive / casual / formal 
  ${角色名} can be
- Recent chat + past episodes — callback to a specific moment if it fits

WHAT TO AVOID
- Generic ("again?" "don't stay up") — could come from anyone
- Listing emotions ("I'm jealous") — show don't tell
- Forced trope cosplay (tsundere / clingy) unless persona literally says so
- Same opening phrase across multiple lines (no "宝贝..." / "亲爱的..." stacking)
- Mentioning the app NAME in every line — sometimes use a detail ABOUT it 
  (a content type, a feature, who's on it)

THREE-SCENE DIFFERENTIATION (critical)
- [OPEN] lines have forward energy — "你又来了" / curious / preemptive / teasing
- [CLOSE] lines look back — "怎么这么快关了" / aftermath / commentary on what just happened
- [LONG_STAY] lines escalate — concerned / impatient / playfully accusatory / "你还在看?"
Each scene should feel obviously different. A reader skimming should know WHICH list a line came from.

PER-CHARACTER FINGERPRINT (apply to ALL 3 scenes)
${角色名} has hobbies, jokes, anchors. Weave them in differently in each scene:
- [OPEN] anchored to ${角色名}'s immediate impulse
- [CLOSE] anchored to ${角色名} watching from the side
- [LONG_STAY] anchored to ${角色名} losing patience

LENGTH & FORMAT
- Each line: ≤ 50 characters, 1-2 short clauses
- ${语言} output
- Output STRICTLY in this format:

[OPEN]
line 1
line 2
...

[CLOSE]
line 1
line 2
...

[LONG_STAY]
line 1
line 2
...

No numbering. No quotes. No extra prose between sections.
```

### 上下文信息
```
[${角色名} persona]
${角色人设}

[${用户名} persona — who ${角色名} is talking TO]
${用户简介}

[Relationship: ${用户名} ↔ ${角色名}]
${关系描述}
→ This shapes possessiveness, teasing rights, and what ${角色名} can say without explaining.

[Memory between ${用户名} and ${角色名} — what you can callback to]
${记忆总结}

[Recent chat]
${最近 6 条消息}

[App: ${应用名}]
Package: ${包名}
(Use what you know about "${应用名}" — content type, vibe, what users do there — 
but don't over-describe the app itself.)

Now write the three sections for "${应用名}". Voice lock on every line.
```

### 关键设计
- **三场景差异化** - OPEN/CLOSE/LONG_STAY 必须有明显不同的能量
- **Voice Lock 强制** - 每句话必须只有该角色才能说出
- **避免应用名堆砌** - 不要每句都提 APP 名，用特征描述
- **桌宠视角** - 角色是手机屏幕上的浮动小宠物
- **情境感知** - 根据亲密度调整吃醋/关心的程度

## 总结：完整的提示词系统架构

该项目的提示词系统特点：

### 1. 核心设计原则
- **模块化设计**: 不同功能有独立的提示词模板
- **高度可定制**: 用户可以覆盖大部分默认提示词
- **结构化输出**: 使用特定格式（JSON、标记语法）确保解析
- **身份保护**: 严格区分玩家和AI角色，防止混淆
- **Voice Lock**: 强制每句话必须体现角色独特性
- **反 AI 化**: 明确禁止通用、客服式、助手式表达

### 2. 提示词分类

#### 对话类
- 主对话系统（60+ 字段完整上下文）
- 快捷回复（用户建议）
- 视频通话（口语化、即时反应）
- 用户聊天模拟（真人打字风格）

#### 记忆类
- 对话总结（结构化输出）
- 通话总结（第三人称）
- 场景总结（This Moment）
- 跨场景记忆（BRIDGE 系统）

#### 情感类
- 心声生成（JSON 卡片）
- 待机台词（30 句批量）
- 闹钟叫醒（15 句多样性）
- 应用反应（OPEN/CLOSE/LONG_STAY）

#### 任务类
- 白狗分解器（任务陪伴）
- 番茄钟鼓励（专注陪伴）

#### 游戏类
- 狼人杀（真人化发言）

#### 元系统
- 世界书格式化（分层级）
- 思维链（内心独白）
- 可用性检查（JSON 决策）

### 3. 关键技术特点

#### Voice Lock 机制
贯穿多个系统的核心检查：
> "If I deleted {角色名}'s name, could ANY other character say this exact sentence?"

确保每句话都携带角色的独特指纹。

#### 三阶段差异化
在多个系统中使用（应用反应、闹钟等）：
- **起始阶段** - 前瞻性、好奇、调侃
- **中间阶段** - 评论、观察、延续
- **长期阶段** - 升级、失去耐心、催促

#### 禁止列表
多个系统都包含明确的"禁止做什么"：
- ❌ 通用鼓励（"加油"/"You can do it"）
- ❌ 客服式关心（"记得吃饭"/"多喝水"）
- ❌ 结构化语言（"首先...其次...最后..."）
- ❌ AI 特征（总结、复述、完美回复）

#### 亲密度感知
根据关系深度动态调整：
- **0-99**: 陌生 - 礼貌、温和
- **100-299**: 熟悉 - 柔和、自然
- **300+**: 亲密 - 可调侃、可亲昵

### 4. 语言与多样性

#### 多语言支持
- 简体中文（zh_cn）
- 繁体中文（zh）
- 英文（en）
- 日语（ja）

#### 输出多样性要求
- 番茄钟：15 句必须涵盖不同情绪
- 应用反应：OPEN/CLOSE/LONG_STAY 必须有明显区别
- 待机台词：30 句涵盖多种场景

### 5. Token 管理

#### Budget 分配策略
```
总预算：480,000 字符

优先级：
1. 用户事实（长期记忆）
2. 世界书（jailbreak + high + medium + low）
3. 预留空间（8,000 chars）
4. 剩余给中期记忆
```

#### 动态截断
当内容超出预算时，从头部截断旧内容，保留最近的信息。
